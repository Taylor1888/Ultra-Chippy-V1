ArtifactFlow

A standalone, artifact-gated control plane for durable multi-agent software-development workflows, optimised for minimal context, high token yield, reproducible evidence, and bounded correction loops.

DocumentProduct Requirements Document
Version0.1
Product typeStandalone control plane
Initial deploymentLocal-first / small team
Core rule: durable, immutable artifacts—not conversation transcripts—are the authoritative workflow state. Stages produce candidate artifacts; independent gates decide whether those artifacts may advance.

1. Executive summary

ArtifactFlow coordinates long-running coding work across specialised agents without carrying an ever-growing conversation through every stage. It decomposes a software-development objective into a durable sequence of specification, slicing, test-contract design, planning, implementation, review, verification, integration, and human acceptance. Each transition is unlocked by explicit evidence.

The product consists of a framework-neutral workflow engine, an immutable artifact registry, deterministic gate evaluators, an event store, a Git/workspace manager, a sandbox runner, a minimal-context broker, and pluggable runtime adapters. Hermes can be one runtime adapter, but the domain model and execution semantics must not depend on Hermes.

The first release is deliberately headless-first. A small operator UI will expose runs, artifacts, gates, evidence, retries, and approvals. An Agent Deck-style visual designer for agents, skills, models, prompts, tool policies, context policies, workflow graphs, gates, budgets, and escalation rules follows only after the engine semantics are proven.

2. Problem and opportunity

2.1 Current problems

  • Long agent conversations accumulate irrelevant reasoning, command output, and stale assumptions, reducing useful context and increasing cost.
  • Agents frequently self-report completion without machine-verifiable evidence.
  • Plans, patches, reviews, and test results are passed as untyped text and lose provenance.
  • Reviewers may inspect workspaces that change while they review, producing race conditions and irreproducible findings.
  • Failed tests are commonly routed back to implementation even when the defect is in the plan, test, specification, slice boundary, or environment.
  • Retries overwrite prior attempts, hide failure history, and can loop without proving progress.
  • Existing agent frameworks are good runtimes but are not necessarily durable workflow control planes.

2.2 Opportunity

A workflow engine that treats every meaningful handoff as a typed, immutable artifact can reduce context size, improve auditability, support crash recovery, and make multi-agent work measurable. By separating workflow control from agent execution, ArtifactFlow can use Hermes, Claude Code, Codex, OpenCode, local agents, deterministic scripts, and future runtimes behind one contract.

3. Goals and non-goals

3.1 Goals

  • GOAL-001 Execute a complete specification-to-release workflow that survives restarts.
  • GOAL-002 Minimise context by handing agents compact stage envelopes plus artifact handles.
  • GOAL-003 Make every transition reproducible from stored state and evidence.
  • GOAL-004 Support specialised profiles, skills, models, prompts, tools, and behavioural policies.
  • GOAL-005 Enforce deterministic validation before semantic LLM review.
  • GOAL-006 Run bounded implementation/review/fix loops with progress detection.
  • GOAL-007 Route failures to the stage responsible for their cause.
  • GOAL-008 Maintain exact requirement-to-release traceability.
  • GOAL-009 Support pluggable runtimes without coupling the engine to a single agent framework.
  • GOAL-010 Measure token yield and workflow effectiveness rather than token count alone.

3.2 Non-goals for MVP

  • Multi-tenant SaaS billing and marketplace functionality.
  • A production-grade drag-and-drop graph designer.
  • Training or fine-tuning foundation models.
  • Replacing Git, CI, test frameworks, or agent runtimes.
  • Automatically promoting to a protected target branch without human approval.
  • Using an LLM to fabricate or infer command/test outcomes.

4. Product principles

  1. Artifacts over transcripts. Transcripts are optional audit detail, not required state.
  2. Candidate production is separate from acceptance. A producer cannot solely approve its own output.
  3. Deterministic facts are owned by software. Hashes, exit codes, Git state, files, and test counts come from runners.
  4. Fresh sessions at semantic boundaries. Planning, implementation, review, verification, and integration normally use independent sessions.
  5. Immutable checkpoints. Reviewers inspect a commit or content-addressed patch that cannot change beneath them.
  6. Append-only attempts. A rejected artifact remains; a correction creates a new version.
  7. Cause-specific routing. Correction returns to the earliest responsible stage.
  8. Bounded autonomy. Every loop has attempt, token, time, cost, and no-progress limits.
  9. Human authority at consequential boundaries. Exact specification acceptance and protected-branch promotion require explicit approval.
  10. Portable core, replaceable runtimes. Workflow semantics remain framework-neutral.

8. Specification-to-release workflow: Gates 0–9

Gate 0 — Project intake and technical baseline

Purpose: establish the factual starting point so pre-existing conditions are not misclassified as regressions.

Activities

  • Validate repository and target branch.
  • Capture base commit and dirty state.
  • Discover build, test, lint, type-check, package, migration, and security commands.
  • Run configured baseline checks.
  • Map major modules, languages, dependencies, and architectural seams.
  • Record environment and sandbox identity.

Artifacts

  • ProjectBaseline
  • RepositoryMap
  • BaselineVerificationRun
  • ToolchainManifest

Pass: repository identity, exact base commit, environment identity, commands, and known baseline failures are recorded. Infrastructure uncertainty blocks operationally rather than advancing.

Gate 1 — Collaborative specification and adversarial clarification

Purpose: convert the user's idea into an explicit, testable specification accepted at an exact version.

Activities

  • Conduct an intensive discovery interview that challenges assumptions and terminology.
  • Clarify actors, behaviours, boundaries, constraints, failure cases, security, privacy, operations, and non-functional needs.
  • Assign stable requirement and decision IDs.
  • Produce a versioned specification candidate.
  • Run an independent red-team review.
  • Route unresolved findings back into clarification.
  • Request explicit user approval of an exact content hash.

Artifacts

  • DiscoveryRecord
  • Specification
  • SpecificationReview
  • DecisionLog
  • HumanAcceptanceDecision

Pass: all blocking review findings are resolved or explicitly excepted; the user approves the exact specification version. Approval never transfers to modified content.

Gate 2 — Vertical-slice design and test-contract creation

Purpose: decompose the accepted specification into independently valuable, testable increments and define observable contracts before implementation planning.

Activities

  • Create stable slice IDs such as SLICE-001.
  • Prefer vertical behaviour through relevant layers over horizontal technical batches.
  • Define seams, dependencies, integration order, and ownership.
  • Trace every slice to requirement IDs.
  • Design acceptance and contract tests independent of an assumed implementation.
  • Independently review tests for coverage, false positives, over-specification, feasibility, and isolation.

Artifacts

  • SliceSet
  • TestContractSet
  • TestContractReview
  • TraceabilityMatrix
  • SliceDependencyGraph

Pass: every accepted requirement is covered or explicitly deferred, slice dependencies form a valid DAG, and test contracts pass independent review.

Gate 3 — Checklist implementation planning

Purpose: create a detailed, executable plan for each slice without performing implementation.

Activities

  • Assign stable plan-item IDs.
  • Identify affected components, prerequisites, dependency order, data changes, migrations, rollout and rollback concerns.
  • Define per-item validation and completion evidence.
  • Trace plan items to requirements, slice IDs, and test contracts.
  • Review feasibility, scope, ordering, and architectural fit.

Artifacts

  • SliceImplementationPlan
  • PlanReview
  • PlanTraceabilityMap

Pass: all required work is checklistable, ordered, traceable, and independently reviewed. Downstream plan defects route back here.

Gate 4 — Bounded Ralph implementation loop with asynchronous A2A review

Purpose: implement one slice through bounded iterations while reviewers assess immutable checkpoints asynchronously.

Iteration envelope

  • Slice definition and relevant plan items.
  • Relevant test contracts.
  • Current repository checkpoint.
  • Previous iteration summary.
  • Outstanding corrective actions.
  • Remaining budgets.

Artifacts

  • ImplementationCheckpoint
  • CheckpointReview
  • IterationSummary
  • CorrectiveActionSet

Each iteration publishes a commit or content-addressed patch before review. Review covers standards, scope, architecture, security, maintainability, and test quality. Review is bound to the exact checkpoint hash and cannot observe a changing worktree.

Pass: required plan items are implemented at an immutable checkpoint, deterministic progress is proven, blocking A2A findings are resolved, and budgets remain within policy.

Gate 5 — Deterministic tests and failure classification

Purpose: establish execution facts in an isolated, reproducible environment and determine the responsible correction stage.

Runner records

  • Exact command and working directory.
  • Exit code and signal.
  • Unaltered stdout/stderr handles.
  • Test counts and duration.
  • Environment and image identity.
  • Repository/checkpoint hash.

Artifacts

  • TestRun
  • FailureDiagnosis
  • CorrectiveActionSet
  • FlakinessAssessment

An agent may diagnose evidence but cannot alter the runner's result. Classifications are implementation, plan, test contract, slice design, specification, dependency/seam, infrastructure, flaky/non-deterministic, or unknown.

Pass: all required deterministic checks pass at the exact checkpoint. Infrastructure failures block operationally and do not count as code failures.

Gate 6 — Independent intent-focused slice review

Purpose: verify that passing code matches the accepted intent rather than merely satisfying tests.

The reviewer receives the accepted specification, slice definition, test contracts, implementation plan, exact checkpoint, deterministic evidence, and outstanding findings. It checks scope, correctness, architecture, security, maintainability, compatibility, migration safety, and test adequacy.

Artifacts: SliceCodeReview with a closed verdict: ACCEPT, REQUEST_CHANGES, ESCALATE_HUMAN, or REJECT_UPSTREAM_DEFECT.

Pass: independent acceptance of the exact checkpoint with no unresolved blocking finding.

Gate 7 — Ordered incremental slice integration

Purpose: integrate accepted slices one at a time in dependency order so conflicts and regressions remain attributable.

Activities

  • Create a versioned integration plan.
  • Merge one accepted checkpoint into an integration branch.
  • Record lineage, conflicts, and resolutions.
  • Run impacted tests after every merge.
  • Return semantic changes to review rather than silently altering accepted output.

Artifacts

  • IntegrationPlan
  • IntegrationStep
  • ConflictResolution
  • IntegratedBuild

Pass: each slice is integrated in topological order with passing evidence and exact commit lineage.

Gate 8 — Full-system verification and release review

Purpose: verify the integrated system and create an immutable release candidate.

  • Run the complete test suite, build/package checks, static analysis, type checks, configured security checks, end-to-end tests, and migration/rollback checks.
  • Audit requirement, slice, test, plan, checkpoint, and evidence traceability.
  • Run an independent final semantic review after deterministic verification.
  • Record known risks, accepted exceptions, and unresolved non-blocking findings.

Artifacts: SystemVerificationRun, TraceabilityAudit, ReleaseReview, and ReleaseCandidate.

Pass: configured verification succeeds, coverage is complete or explicitly excepted, and the final reviewer accepts the exact release commit.

Gate 9 — Human acceptance and protected-branch promotion

Purpose: separate automated verification from the consequential human decision to promote.

The UI presents the exact release candidate, verification evidence, requirement coverage, changes, known risks, exceptions, and human-review flags. The approver may approve, reject, request changes, or accept with documented exceptions if policy allows.

Artifacts: HumanAcceptanceDecision and PromotionRecord.

Pass: an authorised human approves the exact verified commit and ArtifactFlow promotes that same commit to the protected target. Any content change after verification requires re-verification.

9.2 Requirements

  • ART-001 Artifacts are immutable; correction creates a new version.
  • ART-002 Every artifact has a versioned schema and content hash.
  • ART-003 Payload hashes are recomputed on ingestion.
  • ART-004 Producer, inputs, workspace, checkpoint, and evaluator provenance are recorded.
  • ART-005 Large payloads live in blob storage; manifests remain cheap to inject.
  • ART-006 Uniqueness is scoped by run, stage, attempt, kind, and semantic progress—not global hash uniqueness.
  • ART-007 Artifacts support stable requirement, decision, slice, test, plan-item, and finding IDs.
  • ART-008 Artifacts can be exported with schemas, evidence, dependency graph, and event history.

10. Gate engine

10.1 Evaluation order

  1. Presence of required artifacts and fields.
  2. Schema validation against pinned versions.
  3. Integrity: recompute hashes and sizes.
  4. Provenance: active assignment, workspace, producer, and input matching.
  5. Freshness: candidate belongs to the current attempt and active upstream versions.
  6. Progress: relevant state differs from the rejected attempt.
  7. Deterministic verification: commands, policies, tests, and checks.
  8. Semantic review by an independent profile or structured judge.
  9. Human approval where required.
{
  "ok": false,
  "gate": "progress",
  "codes": ["PATCH_UNCHANGED"],
  "compared_to": "art_previous",
  "responsible_stage": "implementation",
  "repair": "Publish a checkpoint with a changed normalized patch hash."
}

10.2 Gate policy requirements

  • Evaluators and schemas are version-pinned per run.
  • A semantic pass cannot override failed deterministic checks.
  • Gate application and downstream scheduling are transactional and idempotent.
  • Every failure returns machine-readable codes, evidence references, responsible scope, and permitted transitions.
  • Producer self-approval is prohibited unless an explicit low-risk policy exception exists; it is never allowed for release promotion.

11.1 Invalidation

When an accepted upstream artifact changes, the engine traverses dependency edges and marks affected descendants STALE. Stale status records the triggering artifact, prior accepted version, new version, rule used, and required re-evaluation. Unaffected branches may continue.

  • Requirement change may invalidate slices, tests, plans, implementations, reviews, integrations, and releases.
  • Test change may invalidate plan coverage, implementation acceptance, and verification evidence.
  • Implementation checkpoint change always invalidates reviews and test evidence bound to the old hash.
  • Integration conflict resolution that changes semantics invalidates slice acceptance.
  • Release-candidate mutation invalidates human approval.

12.2 Context policy

  • Start fresh sessions at semantic boundaries.
  • Do not inject full upstream transcripts or historical tool output.
  • Provide artifact manifests and summaries first; retrieve payloads on demand.
  • Expose artifact_get, artifact_search, stage_inputs, source-search, and scoped file-read tools.
  • Track every retrieval and repeated retrieval of the same object.
  • Allow bounded continuity only within an implementation/fix loop where local working memory has demonstrated value.
  • Generate an iteration summary before context reset; it is an artifact and may not claim unverified facts.

12.3 Token-yield metrics

  • Input/output tokens per accepted artifact and successful stage.
  • Accepted implementation change per coding turn.
  • Defects found per review token.
  • Repeated artifact/source retrieval.
  • No-progress attempts and compression events.
  • Cost of semantic judges versus prevented wasted work.
  • Cache hit/stability within a stage.

13.2 Runtime adapter interface

  • capabilities()
  • create_session(profile, envelope, workspace)
  • send(session, message)
  • stream_events(session)
  • request_artifact(session, contract)
  • usage(session)
  • cancel(session)
  • resume(session) where supported

The adapter normalises tool events, usage, completion, failure, and artifact submission but does not own stage transitions. Initial adapters should include a deterministic fake runtime and one real runtime. Hermes is a strong first real adapter; additional adapters can target Claude Code, Codex, OpenCode, or a generic OpenAI-compatible agent.

14. Git, checkpoints, and workspace management

  • Capture repository URL/identity and base commit at Gate 0.
  • Use isolated worktrees or equivalent workspaces per concurrent writable stage.
  • Every implementation iteration publishes an immutable commit or content-addressed patch.
  • Record base/head commit, normalized patch hash, changed files, authoring stage, and workspace.
  • Reviewers receive read-only access to the exact checkpoint.
  • Integration occurs on a dedicated branch one slice at a time.
  • Conflict resolution is a new artifact with provenance and may require renewed review.
  • Protected target promotion is limited to the exact Gate 8 release commit after Gate 9 approval.
  • Workspace cleanup never deletes the authoritative checkpoint or artifact record.

15. Deterministic sandbox execution

  • Commands run in an isolated environment selected by project policy.
  • Record image/environment digest, OS/runtime versions, variables allowlist, working directory, command, start/end time, exit code, signal, and logs.
  • Secrets are injected by reference and redacted from stored output.
  • Network policy is configurable per command/stage.
  • Timeout, CPU, memory, disk, process, and output-size limits are enforced by software.
  • Test reports may be parsed into structured counts, but original logs remain immutable.
  • Agents may explain results but cannot create a passing TestRun without runner evidence.
  • Equivalent reruns can be compared to identify flakiness and environmental drift.

16. Loops, budgets, and no-progress detection

BudgetExamples

Tokens/costPer stage, per slice, per run, and per runtime/model.

Budget exhaustion creates a durable WAITING_FOR_HUMAN or FAILED_BUDGET state with complete attempt history, unresolved findings, consumed budget, and available decisions. It never silently extends itself.

17. Persistence, idempotency, and recovery

17.1 Storage

  • Relational database for runs, stages, attempts, artifacts, gates, edges, leases, approvals, and projections.
  • Append-only event table for audit and replay.
  • Content-addressed local/object blob store for large payloads and logs.
  • Git repositories/worktrees for source state.

17.2 Events

At minimum: run created; stage scheduled, claimed, started, heartbeat, completed, blocked or failed; artifact submitted, rejected, accepted or invalidated; gate evaluated; correction routed; transition selected; human decision recorded; integration completed; release promoted; run completed or cancelled.

17.3 Reliability requirements

  • Transition application and child scheduling are transactional.
  • Task creation and artifact submission use idempotency keys.
  • Workers use leases and heartbeats; expired work is safely reclaimable.
  • Duplicate callbacks do not duplicate artifacts or downstream tasks.
  • After restart, materialised state can be rebuilt from events and durable domain records.
  • Unknown execution outcome is represented explicitly and reconciled before retry.

  • Run graph: stages, attempts, accepted/rejected/stale artifacts, transitions, and correction routes.
  • Artifact viewer: schema-aware rendering, hash, provenance, dependencies, version diff and payload download.

21. Security and secrets

  • Least-privilege tools and workspace permissions per profile and stage.
  • Separate read, write, test, merge, approve, and promote capabilities.
  • Secrets stored outside prompts/artifacts and injected by opaque reference.
  • Secret and credential-like values redacted from logs, events, reviews, and exports.
  • Untrusted repository content is treated as data, not control instructions.
  • Sandbox network and filesystem scope is explicit and auditable.
  • Human approvals bind to authenticated identity, exact hash, policy version, time, and rationale.
  • Protected target credentials are available only to the promotion service after approval.
  • Artifact schemas and uploads enforce size, path, MIME, archive, and traversal controls.
  • Audit records are append-only and tamper-evident.

22. Observability and audit

22.1 Metrics

  • Stage duration, queue delay, retries, failure classes and gate pass rate.
  • Tokens, cost and accepted-artifact yield by stage/profile/model.
  • No-progress rate, repeated retrieval and context compression.
  • Defects discovered by checkpoint review, tests, slice review and release review.
  • Invalidation fan-out and rework caused by each upstream stage.
  • Worker/runtime health, lease expiry, sandbox failures and command duration.
  • Human wait time and decision outcomes.

22.2 Audit export

A run export contains definitions and versions, artifact manifests and payload references, dependency graph, gate evaluations, deterministic evidence, checkpoint lineage, human decisions, event stream, usage/cost, and redaction report. Export must be sufficient to explain why every transition occurred without requiring private chain-of-thought.

24. Validation and test strategy

24.1 Deterministic fake-agent simulation

Build a scripted fake runtime before relying on live models. Scenarios must cover:

  • Happy path from project creation to human promotion.
  • Candidate artifact rejection followed by corrected version.
  • Duplicate submission and duplicate callback.
  • Crash before and after artifact commit, gate commit, and downstream scheduling.
  • Expired lease and worker reclamation.
  • Unchanged patch/no-progress attempt.
  • Stale review arriving after a newer checkpoint.
  • Specification revision invalidating descendants.
  • Test, plan, slice, specification, infrastructure, flaky, and unknown failure routing.
  • Integration conflict and semantic conflict resolution.
  • Budget exhaustion and human escalation.
  • Human approval of the wrong hash rejected.
  • Promotion attempt of a commit other than the verified release rejected.

24.2 Test layers

  • Schema and property tests for artifacts and definitions.
  • State-machine and event-replay tests.
  • Transactional/idempotency tests with fault injection.
  • Runtime-adapter contract tests.
  • Git/worktree and sandbox integration tests.
  • Security, permission and redaction tests.
  • End-to-end fake-agent workflows.
  • Controlled live-model evaluations measuring completion, correctness, token yield and recovery.

24.3 Live-model evaluation

Use a fixed suite of small repositories and tasks with known acceptance tests. Compare models and context policies on accepted artifacts, escaped defects, retries, wall time, cost, tokens, repeated retrieval, and human interventions. Model output is never the benchmark oracle; deterministic tests and reviewed expected artifacts are.

26. MVP acceptance criteria

  • A run can execute Gates 0–9 using the fake runtime and at least one real runtime adapter.
  • The process can be killed and restarted during every stage without losing accepted state or duplicating side effects.
  • Every downstream stage is demonstrably gated by typed, validated artifacts.
  • Rejected artifacts remain immutable and corrected versions retain lineage.
  • Review and test evidence bind to exact checkpoint hashes.
  • Deterministic failures route to implementation, planning, tests, slicing, specification, infrastructure, flakiness, or human triage as configured.
  • Upstream revision automatically marks affected descendants stale and prevents their use.
  • Implementation/review loops stop on approval, blocker, no progress, or budget exhaustion.
  • Accepted slices integrate incrementally with tests after each merge.
  • Human approval is required for the exact release candidate before protected-target promotion.
  • The UI exposes run state, graph, artifacts, gates, evidence, checkpoints, budgets, events, blockers, and approvals.
  • An audit export explains every transition without relying on full conversation transcripts.
  • Token and cost metrics are available per stage and accepted artifact.
  • No secret value is persisted in an artifact, prompt log, event payload, or audit export.

27. Product decisions and open questions

27.1 Decisions already made

  • Standalone workflow control plane with pluggable runtime adapters.
  • Workflow engine before visual designer.
  • Artifacts and events are authoritative; transcripts are secondary.
  • Immutable artifact versions and independent acceptance.
  • Deterministic validation precedes semantic review.
  • Fresh sessions at semantic boundaries.
  • A2A review targets immutable checkpoints.
  • Failure routing is cause-specific.
  • Descendant invalidation is automatic.
  • Integration is incremental and topological.
  • Protected-branch promotion requires human acceptance.

27.2 Open questions for implementation planning

  • Preferred implementation language and database for the control plane.
  • Whether local blob storage plus SQLite is sufficient for MVP or PostgreSQL/object storage should be used immediately.
  • Initial UI stack and whether CLI/TUI operation precedes web UI.
  • First real runtime adapter after the fake runtime.
  • Sandbox technology and cross-platform support expectations.
  • Authentication model for local-first and small-team deployments.
  • Policy language: typed YAML/JSON, embedded expression language, or constrained code plug-ins.
  • Retention, archival, and deletion policy for transcripts, logs, worktrees, and blobs.
  • Whether semantic evaluators require consensus, model diversity, or configurable quorum for high-risk gates.
  • Exact rules for accepted exceptions and who may authorise them.
